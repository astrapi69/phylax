# Phylax Store Deployment Guide

How to publish Phylax to the major application stores. Phylax is a
PWA, so most stores accept it via a PWABuilder-generated wrapper;
Apple App Store needs a manual Capacitor / Xcode step. None of this
flow is automated; the runbook below is the maintainer's
canonical reference.

The PWABuilder-specific mechanics are in
[`d-03-pwabuilder.md`](d-03-pwabuilder.md). This file is the
umbrella store-deployment guide.

## Per-store workflow

### Microsoft Store (Windows + Xbox)

**Format:** `.msixbundle` (signed PWA wrapper).
**Path:** PWABuilder → "Package for Stores" → Windows → fill in
publisher details → download.

1. Follow [`d-03-pwabuilder.md`](d-03-pwabuilder.md) §2 Microsoft
   Store path.
2. Upload `.msixbundle` to Partner Center → create new app →
   Submit.
3. Review usually completes in 24-72 hours.

**Re-submission:** bump `package.json` version, regenerate the
bundle, upload as a new submission. Microsoft auto-detects the
version increment.

### Google Play (Android)

**Format:** `.aab` Android App Bundle wrapping a Trusted Web
Activity (TWA).
**Path:** PWABuilder → "Package for Stores" → Android → fill in
identity + start URL → download `.aab` + Digital Asset Links JSON.

1. Follow [`d-03-pwabuilder.md`](d-03-pwabuilder.md) §2 Google Play
   path.
2. Place the DAL JSON at
   `https://astrapi69.github.io/.well-known/assetlinks.json` -
   this lives in the `astrapi69.github.io` repo (org / user
   pages), NOT in the Phylax repo, because the path is on the
   apex host.
3. Upload `.aab` to Play Console → internal testing track →
   verify DAL → promote to production.

**Re-submission:** bump version code (integer) AND version name in
the TWA build before regenerating. Play Console rejects duplicate
codes.

### Samsung Galaxy Store (Android Samsung devices)

**Format:** PWA URL submission OR `.apk` / `.aab`.
**Path:** Samsung Galaxy Store accepts direct PWA URL submissions

- the simplest of the three Android-side stores.

1. Sign in at https://seller.samsungapps.com/.
2. New App → "Web App" → submit
   `https://astrapi69.github.io/phylax/`.
3. Provide app icon + screenshots + descriptions.
4. Samsung's review tends to focus on responsive layout + offline
   support; both are covered by the existing PWA infrastructure.

**Re-submission:** edit the existing app entry, no version bump
needed for URL-based submissions.

### Apple App Store (iOS / iPadOS)

**Format:** native iOS app (`.ipa`) - Apple does NOT accept raw
PWA submissions.

Two viable paths:

#### Option A: PWABuilder iOS package

1. PWABuilder → "Package for Stores" → iOS.
2. Download the generated Xcode project.
3. Open in Xcode (macOS required). Configure signing identity +
   bundle ID (`com.asterios.phylax`).
4. Update the WKWebView wrapper's start URL to point at the
   live D-01 deployment.
5. Test on a physical iOS device (Apple still rejects iOS apps
   that have only been simulator-tested).
6. Archive → Upload to App Store Connect → Submit for review.

#### Option B: Capacitor wrapper

1. Run `npm i @capacitor/core @capacitor/cli @capacitor/ios` in a
   fork of the Phylax repo (do NOT pollute main package.json -
   Capacitor is iOS-only and pollutes the dependency surface).
2. `npx cap init Phylax com.asterios.phylax`.
3. `npx cap add ios`.
4. Build the production bundle (`make build`) and copy `dist/`
   into the Capacitor iOS project as the web layer.
5. Same Xcode submission flow as Option A from step 3 onward.

**Apple-specific gotchas:**

- Apple rejects PWAs that "merely link to a web app" without
  added value. The current Phylax build offers
  install-as-a-PWA already; the App Store version needs a
  meaningful difference (e.g., share-extension, file-handler).
  Consider implementing Web Share Target + Open-with handlers
  before the first iOS submission.
- Apple's privacy nutrition label has to match the in-app
  disclaimer wording verbatim. Multi-AI-Provider (ADR-0019)
  means listing all seven providers as "third-party
  optionally-engaged services".
- IndexedDB quota on iOS is 7 days unless the user explicitly
  installs to home screen; this is true on both PWA and native
  paths and has to be disclosed.

**Re-submission:** Apple rejects build numbers that already
exist in App Store Connect. Bump CFBundleVersion in the Xcode
project on every submission.

### Amazon Appstore (Fire devices, Android sideload)

**Format:** `.apk` or `.aab` (TWA).

1. Reuse the PWABuilder `.aab` from the Google Play path.
2. Sign in at https://developer.amazon.com/.
3. New app → upload bundle → fill in metadata.
4. Amazon's review is usually fast (24-48 hours) but their
   automated test suite sometimes flags TWAs as "not
   responsive" due to viewport-rendering quirks; manual
   testing on a Fire tablet helps catch these.

### F-Droid (FOSS Android repository)

**Format:** native build from source (FOSS-only).

F-Droid does NOT accept TWAs because the Trusted Web Activity
binary contains a closed-source Chrome runtime. Submission
requires either:

- Accept the FOSS-purity exception (some apps publish on
  F-Droid as TWAs anyway, marked as "anti-feature: NonFreeNet"),
  OR
- Ship a Capacitor build that bundles a FOSS WebView (Bromite
  WebView, etc.) - significant additional engineering.

Recommended: defer F-Droid until either the FOSS exception is
relaxed or a Capacitor-with-Bromite build is engineered. For now,
F-Droid is "out of scope".

## Submission cadence

- **Major release** (`x.0.0` SemVer bump): submit to every active
  store. Update screenshots if UI surface changed.
- **Minor release** (`x.y.0`): submit only if a feature added is
  user-visible enough to motivate a re-review (most Phase 7
  polish doesn't qualify; AIP-01..05 multi-provider does).
- **Patch release** (`x.y.z`): skip store submissions; the PWA
  auto-updates via the silent-update SW (BUG-01) and that
  reaches installed users without store-side action.

## Post-submission monitoring

- Microsoft Store: Partner Center dashboard alerts on review
  failures.
- Google Play: Console flags integrity issues; the most common
  is the DAL file becoming unreachable (HTTP 404 if the
  `astrapi69.github.io` repo path drifts).
- Apple App Store: review feedback arrives via App Store
  Connect notifications.
- Samsung Galaxy Store: Seller Office surface; review feedback
  is rare for URL-based PWA submissions.

## Why not automate?

- Store credentials cannot live in the repo or CI. Storing a
  Microsoft Partner Center service principal or an Apple
  certificate in GitHub Actions exposes a high-value attack
  surface for a single-maintainer project.
- Submission decisions (track, rollout %, store-policy
  compliance) are human-judgement calls.
- Each store's review feedback is non-deterministic; an
  automation that re-submits on rejection can quickly hit a
  store-side rate limit (Apple in particular).
- Phylax's local-first architecture
  (`.claude/rules/architecture.md`) explicitly avoids backend
  services; an automated submission pipeline contradicts that
  posture.
