# Architecture Realignment After Medium Series Review

**Audit date:** 2026-04-12

## 1. What the old CONCEPT and ROADMAP assumed

The current `docs/CONCEPT.md` and `docs/ROADMAP.md` model Phylax as a structured health diary with five rigid entry types: symptom, medication, vital, appointment, and note. Each entry type has its own form, list view, and detail view. The data model is record-oriented: individual entries stored in an `entries` table, each encrypted independently.

Key assumptions of the old model:

- **Entry types as equal citizens.** All five types share a generic `entries` table with a `type` discriminator. CRUD operations are per-type. The UI is organized around creating, listing, and filtering entries.
- **AI as optional Phase 7.** The AI assistant is the last phase, explicitly opt-in, and separated from the core product. Phases 1-6 forbid all external API calls. The AI flow is a chat that produces draft entries.
- **No canonical output document.** There is no concept of a unified profile document. Entries exist as individual records. PDF export (Phase 4) aggregates entries into a report, but the report is a generated artifact, not the primary data structure.
- **Single flat timeline.** Entries are sorted by date in a unified timeline view. Grouping is by date range and type filter, not by health theme.
- **No version history of the profile.** Individual entries have `createdAt`/`updatedAt`, but there is no concept of the overall profile evolving as a versioned document.

## 2. What the articles actually describe

The four-part "Lebende Gesundheit" series defines a fundamentally different model.

### The living health profile as primary artifact

The central output of Phylax is not a collection of entries but a **living markdown document** -- a structured, versioned medical profile that grows over time. The profile is the unit of work, the unit of export, and the unit shared with doctors.

### Observation as the core unit (not entry types)

Every health fact is an **Observation** with three facets:

- **Beobachtung (Fact):** What concretely happened or what the data shows.
- **Muster (Pattern):** What recurs -- temporally, situationally, physiologically. Includes links to related observations.
- **Selbstregulation (Decision):** What action was taken as a consequence.

Observations are grouped by **theme** (e.g., "Schulter", "Ernährung", "Blutdruck", "Mobilität"), not by rigid type. A theme can contain subjective observations, lab values, medication context, and lifestyle factors simultaneously.

### Profile template (from Article 3 appendix)

The canonical structure is:

1. **Basisdaten** -- name, age, diagnoses, medications, limitations, managed-by (for proxy profiles)
2. **Blutwerte** -- lab values with date, parameter, result, reference range, contextual assessment
3. **Beobachtungen** -- grouped by theme, each with Beobachtung/Muster/Selbstregulation/Status
4. **Ergänzungsplan** -- supplement/medication schedule with purpose
5. **Offene Punkte** -- checklist for next doctor visit, things to monitor, missing information
6. **Versionshistorie** -- date, what changed, source of information

### AI as primary input method

AI is not an optional add-on. It is the primary way a user creates and updates their profile. The workflow described across all four articles:

1. User provides fragments (lab photos, medication packages, verbal observations)
2. AI structures these fragments into the profile template
3. AI marks uncertainties: `[unklar]`, `[nachfragen]`, `[nur Angehörigen-Perspektive]`
4. AI identifies gaps (missing tests, incomplete data)
5. User reviews the structured output and commits it to their profile
6. Profile is versioned with each update

The AI operates under a strict **prompt contract**: it structures, it does not diagnose. It asks questions, it does not give answers. It marks what it does not know.

### Fallback: manual and paste-in modes

Users who do not want to use AI (or cannot) have two fallback paths:

- Manual form-based entry (the old CRUD flow, now secondary)
- Paste-in: user runs an AI session externally (ChatGPT, Claude) and pastes the resulting markdown into Phylax

### Proxy profiles

Article 3 introduces the **Stellvertreterprofil** (proxy profile) for people who cannot manage their own health (elderly with dementia, children, people with disabilities). The data model must support this from day one via a `profileId` field, even though the MVP is single-profile.

### Version history is first-class

The profile is not a snapshot. It is a living document with explicit version history (1.0, 1.1, 1.2...). Each version records what changed and the source of the change (lab report, doctor visit, caregiver observation).

## 3. Concrete deltas between old and new

| Aspect                  | Old model                                                      | New model                                                                           |
| ----------------------- | -------------------------------------------------------------- | ----------------------------------------------------------------------------------- |
| **Core entity**         | Entry (5 types: symptom, medication, vital, appointment, note) | Observation (3 facets: fact, pattern, self-regulation) grouped by theme             |
| **Primary artifact**    | Collection of individual entries                               | Living versioned markdown profile document                                          |
| **Grouping**            | By type, filtered by date                                      | By health theme (Schulter, Ernährung, Blutdruck, etc.)                              |
| **AI role**             | Optional Phase 7, off by default                               | Primary input method from Phase 2, guided by prompt contract                        |
| **AI timing**           | After all other features are built                             | Early, as the main profile creation workflow                                        |
| **External API calls**  | Forbidden in MVP (phases 1-6)                                  | Allowed for user-initiated AI with user-provided API key                            |
| **Chat persistence**    | AI chat messages stored                                        | Chat messages ephemeral, only structured output saved                               |
| **Output format**       | PDF report generated from entries                              | Markdown profile as canonical form, PDF as secondary export                         |
| **Lab values**          | Vital entry type with single value                             | Dedicated profile section with date, parameter, result, reference, assessment       |
| **Version history**     | Per-entry createdAt/updatedAt                                  | Profile-level version history with change log and source                            |
| **Proxy support**       | Not mentioned                                                  | Structurally supported from day one via profileId, MVP is single-profile            |
| **Uncertainty markers** | Not present                                                    | First-class: [unklar], [nachfragen], [nächste Kontrolle], [Ärztin besprechen]       |
| **Supplement plan**     | Part of medication entries                                     | Dedicated profile section with time-of-day schedule                                 |
| **Open points**         | Not present                                                    | Dedicated checklist section for next doctor visit                                   |
| **Manual CRUD**         | Primary input method                                           | Secondary fallback for quick additions                                              |
| **Dexie schema**        | entries, documents, meta tables                                | profiles, observations, lab_values, supplements, open_points, documents, meta (TBD) |

## 4. Recommendations for realignment tasks

### R-01: Update CONCEPT.md

Rewrite `docs/CONCEPT.md` to reflect the living health profile vision. Replace the five entry types with the observation model. Define the profile template as the canonical data structure. Document the AI-as-primary-input workflow. Add the proxy profile concept. Keep the threat model and crypto approach unchanged.

### R-02: Restructure ROADMAP.md

Reorganize the phases:

- **Phase 1 (Foundation):** Unchanged. Crypto, PWA, key management, testing infrastructure. F-08 through F-18 remain valid (though F-10 schema and F-11 repository need new entity definitions).
- **Phase 2 (Profile):** Replace "Entries CRUD" with profile creation. Core profile document structure, observation CRUD grouped by theme, profile versioning, markdown rendering and export.
- **Phase 3 (AI-guided input):** Move AI from Phase 7 to Phase 3. API key management (encrypted), prompt contract implementation, guided profile session, ephemeral chat, structured output review and commit. Paste-in mode as fallback.
- **Phase 4 (Lab values and supplements):** Lab value section with contextual assessment, supplement plan with schedule, linking to observations.
- **Phase 5 (Documents):** Unchanged in scope. Upload, encrypt, link to profile sections.
- **Phase 6 (Export and sharing):** PDF export of the profile, CSV export, date range filtering. The markdown IS the canonical form; PDF is a rendered view.
- **Phase 7 (Backup):** Previously Phase 5. Encrypted backup/restore.
- **Phase 8 (Polish):** Mobile-first, i18n, dark mode, accessibility, onboarding.

### R-03: Update README.md

Update the "What is Phylax" section to describe the living health profile vision. Update the "Not a medical device" section to add "AI structures, it does not diagnose." Update the origin section if needed. Update the tech stack to mention AI integration as a core feature (user-provided API key). Update the current status section.

### Rules changes (folded into R-02 or separate commit)

- **ai-workflow.md:** Relax the "no external API calls in MVP" rule for user-initiated AI with user-provided API key. Add the prompt contract as a reference.
- **architecture.md:** Update the data model section. Replace entries with observations and profile sections. Add profileId requirement.
- **CLAUDE.md:** Update the project description and data model references.
