# Manual Accessibility Sweep — 2026-04-29

P-07 audit. One-time manual sweep of the four layers axe smoke does
not catch: keyboard navigation, screen-reader semantics, focus
management, and reduced-motion preferences. Records the v1.0.0-era
a11y posture; findings register as polish tasks for individual
follow-up.

## Method

- **Automated baseline already shipped** (T-02a / T-02b): every
  authenticated screen runs `@axe-core/playwright` against WCAG 2 A
  + AA; the production-E2E suite asserts zero violations.
  `tests/e2e-production/a11y-exclusions.ts` documents intentional
  skips (currently empty). EmptyStatePanel CTA contrast was caught
  by axe and fixed in `ca6cb06`.
- **What this audit covers**: keyboard-only navigation flow, screen-
  reader landmark + label semantics, focus management across modals
  and route changes, reduced-motion compliance, manual contrast spot-
  checks beyond axe.
- **Hardware**: developer machine, Chrome DevTools. Screen-reader
  tier (NVDA / VoiceOver / Orca / TalkBack) NOT exercised in this
  pass — deferred per Q-lock (no SR currently installed). SR layer
  becomes its own follow-up audit when an SR is available.
- **Routes walked**: `/welcome`, `/privacy`, `/setup`, `/unlock`,
  `/profile`, `/observations`, `/lab-values`, `/supplements`,
  `/open-points`, `/timeline`, `/documents`, `/import`, `/chat`,
  `/settings`, `/license`, `/backup/import/select`,
  `/backup/import/unlock`. Modals exercised on each: forms,
  ConfirmDialogs, ResetDialog, NavDrawer, ExportDialog, ExportPreview.

## Findings

Each finding has a severity (high / medium / low), a route or
component, a description, and a recommended fix. High-severity
items become individual polish tasks; medium / low go on the polish
queue without dedicated tasks.

### Keyboard navigation

- **No findings.** Tab order on every screen follows the visual
  reading order. Skip-link is absent (single-page app + always-
  visible main landmark; not flagged as a violation by axe nor
  surfaced as a usability issue in informal testing). All
  interactive elements have a visible focus ring (`focus-visible`
  Tailwind utility consistently applied).
- **Watch-point (low)**: NavDrawer focus trap is sourced from the
  Modal primitive's `useFocusTrap`. Confirmed by the BUG-02
  NavDrawer.test.tsx case "click outside drawer" + Tab cycling.
  Manual tab cycling matches.

### Focus management

- **No findings.** Modal primitive (`src/ui/Modal/`) handles trap
  + return-focus + scroll-lock; every form modal across O-10..O-20
  uses it. `useReturnFocus` consistently restores focus to the
  trigger element after close.
- **Watch-point (low)**: route changes do not move focus to the
  new page's heading. Standard SPA caveat; SR users with no
  heading-jump habit may stay on the previous focus target.
  Adding an `useFocusOnRouteChange()` hook is a future polish but
  not flagged as a violation today.

### Screen-reader semantics (manual review without SR)

- **Finding (medium): `<pre>` in LicenseView lacks an explicit
  `tabindex="0"`** so SR users may not be able to scroll the
  scrollable code-style block via keyboard alone. The block
  contains static legal text so reading order via SR is fine, but
  scrolling within the `<pre>` for users with high-magnification
  setups would benefit from a focusable wrapper. **Polish task**:
  P-07-a — add `tabindex="0"` to the `<pre>` block + verify with
  SR when available.
- **Finding (low): NavDrawer's `<h2>` title duplicates the role**
  ("Navigation"). The drawer is already announced as
  `role="dialog" aria-modal="true" aria-labelledby={...}`; the
  visible title-bar text is redundant for SR. Acceptable as is —
  visual users still benefit. **No action.**
- **Finding (low): ExportDialog format-buttons announce only the
  format name** ("Markdown", "PDF", "CSV") without context that
  these are export-format options. The surrounding heading
  "Format wählen" provides context but it is a sibling, not a
  parent. **Polish task**: P-07-b — wrap the format buttons in a
  `<fieldset><legend>` (or `role="radiogroup" aria-labelledby`)
  for explicit grouping semantics.
- **No high-severity findings.** ARIA labels are consistently
  applied on icon-only buttons; `role="status"` + `aria-live` on
  loading + counter regions; `role="alert"` on error banners.

### Reduced motion

- **Finding (medium): observation match-highlight scrollIntoView
  uses `behavior: 'smooth'`** unconditionally
  (ObservationsView.tsx P-19 implementation). Users with
  `prefers-reduced-motion: reduce` should see an instant scroll
  instead. **Polish task**: P-07-c — gate the smooth-scroll
  behaviour on `window.matchMedia('(prefers-reduced-motion: reduce)')`.
- **Finding (low): NavDrawer slide-in lacks an explicit transition,
  so reduced-motion preference is honoured by default** (no
  animation to disable). No action.
- **Skeleton + ListSkeleton** use `motion-safe:animate-pulse`,
  which already respects `prefers-reduced-motion`. **No action.**

### Manual contrast spot-checks (beyond axe)

- **Finding (low): blue-600 on light gray-50 background** in some
  link contexts (Header brand link on AppShell background) measures
  at 6.4:1 — passes AA, fails AAA on small body text. Per
  `.claude/rules/quality-checks.md` Phylax targets WCAG AA, not AAA.
  **No action.**
- Otherwise all spot-checked combinations pass AA. Axe handles
  this comprehensively already.

### Touch targets

- All major interactive elements verified at 44x44 minimum:
  - SearchInput X clear button (already enforced)
  - NavDrawer hamburger trigger + drawer X + nav links
  - AutoLockSection preset buttons
  - Header lock + ThemeToggle
  - Modal close + footer buttons via the primitive
- **Finding (medium): a few inline NavLinks in NavBar (desktop
  side panel) are ~36 px high** at the default text-sm size.
  Desktop pointer use is the primary case so this is acceptable
  per WCAG 2.5.5 AA exception ("user-agent default size") but
  flagged for future audit. **No immediate action**; logged here.

## Polish queue

The following polish tasks register as ROADMAP follow-ups under
their own IDs:

- **P-07-a**: `<pre>` in LicenseView needs `tabindex="0"` for
  keyboard-scrollable focus.
- **P-07-b**: ExportDialog format buttons need `<fieldset><legend>`
  or `role="radiogroup"` grouping.
- **P-07-c**: Observation match-nav scrollIntoView should honour
  `prefers-reduced-motion: reduce`.

These three are individually small (~5-15 LOC each) and ship as
separate commits when picked up. None are user-blocking; axe smoke
remains the gating bar, this audit registers polish targets only.

## Conclusion

Phylax's a11y posture is solid for v1.0.0:

- Zero high-severity findings.
- Three medium-severity polish targets registered (P-07-a/b/c).
- Screen-reader tier deferred until SR available; surface again as
  P-07-d (separate audit) when ready.

P-07 closes; the polish queue continues independently.

Re-run this audit when:

- A new shared UI primitive lands (Toast / Snackbar would re-
  open form-error a11y questions).
- A user reports a real a11y blocker.
- Screen reader is available for the SR-tier sweep (P-07-d).
